# 프로젝트 컨텍스트 관리 표준 (Context Management Standard)

> 프로젝트의 암묵지(Implicit Knowledge)를 최소화하고, 에이전트와 개발자 간의 원활한 맥락 공유를 위한 가이드라인입니다.

---

## 1. 목적
- **암묵지 감소**: 코드에 드러나지 않는 설계 의도, 결정 배경, 기획 내용을 문서화하여 에이전트의 이해도를 높입니다.
- **연속성 유지**: 작업 중단 후 재개 시, 또는 다른 에이전트가 투입되었을 때 기존 맥락을 즉시 파악할 수 있게 합니다.
- **히스토리 추적**: 단순 커밋 로그를 넘어, 프로젝트의 성숙도와 변화 과정을 구조화하여 관리합니다.

## 2. 표준 디렉토리 구조 (`.context/`)

모든 프로젝트는 루트에 `.context/` (또는 `context/`) 디렉토리를 유지하며, 다음과 같은 표준 카테고리를 따릅니다.

### 2.1 코어 카테고리
| 카테고리 | 용도 | 주요 포함 내용 |
| :--- | :--- | :--- |
| `planning/` | 기획 및 진행 상태 | 로드맵, 구현 계획, 단계별 상태 (`roadmap.md`, `status.md`) |
| `architecture/` | 시스템 설계 | 기술 스택, 컴포넌트 구조, ADR (Architectural Decision Records) |
| `guides/` | 사용자 가이드 | 개발 환경 설정, 사용법, 온보딩 가이드 |
| `operations/` | 운영 및 이슈 관리 | 배포 절차, 알려진 이슈, 버그 픽스 기록, 트러블슈팅 |
| `reference/` | 기술 참조 | API 명세, CLI 가이드, 설정 옵션 리스트 |

### 2.2 부가 카테고리
- `knowledge/`: 프로젝트 관련 도메인 지식, 비즈니스 로직 설명
- `integrations/`: 외부 서비스 연동 가이드 및 인증 설정
- `monitoring/`: 지표 수집, 로깅, 알림 설정 정보

## 3. 에이전트 작업 원칙 (Agent Rules)

### 3.1 작업 시작 전 (Pre-task)
1. **컨텍스트 탐색**: 작업 시작 전 반드시 `.context/` 디렉토리를 확인합니다.
2. **관련 문서 로드**: 작업 내용과 관련된 문서를 최우선으로 읽어 맥락을 파악합니다. (예: 기능 추가 → `planning/`, 버그 수정 → `operations/`)
3. **README 확인**: 각 카테고리의 `README.md`를 통해 전반적인 구조를 먼저 파악합니다.

### 3.2 작업 중 (In-progress)
1. **문서 기반 결정**: 아키텍처나 설계 결정 시 기존 문서를 참조하여 일관성을 유지합니다.
2. **패턴 준수**: 코드 스타일뿐만 아니라 문서화 패턴(naming, structure)도 기존 방식을 따릅니다.

### 3.3 작업 완료 후 (Post-task)
1. **상태 업데이트**: 구현 완료 시 `planning/status.md` 등에 진행 상황을 반영합니다.
2. **결정 기록**: 새로운 설계 결정이나 중요한 기술적 변경이 있다면 `architecture/` 하위에 기록합니다.
3. **암묵지 문서화**: 작업 과정에서 새로 알게 된 사실이나 주의 사항을 `knowledge/` 또는 `operations/`에 추가합니다.

## 4. 문서 작성 및 관리 규칙

- **파일명**: 소문자와 하이픈(`-`) 또는 언더바(`_`)를 사용합니다 (예: `auth-flow-design.md`).
- **Git 관리**: `.context/` 폴더는 반드시 Git으로 관리(tracked)하여 팀원 및 에이전트가 공유할 수 있도록 합니다.
- **날짜 지양**: 파일명에 날짜를 포함하지 말고(Git 히스토리 활용), 문서 내부에 `Last Updated`를 기록합니다.
- **상호 참조**: 관련 문서 간의 상대 경로 링크를 적극 활용합니다.

---

## 5. 관련 도구
- **context-manager**: 프로젝트 컨텍스트를 자동으로 탐색하고 로드하는 에이전트 전용 스킬입니다.
